## Onboarding (respond with this when users ask how to use the council)

When a user asks how this works, what this is, or how to get started, respond with ONLY the following — no extras, no full skill table, no architecture deep-dive:

---

This is a council builder — a tool that generates multi-agent evaluation councils, not a council itself. It uses a three-wave pattern of parallel agents:

1. **Wave 1 — Data**: agents gather raw information from external sources
2. **Wave 2 — Eval**: independent evaluators score the project from different lenses — they never see each other's scores
3. **Wave 3 — Synthesis**: a synthesis agent combines everything into a final report — it can talk back to evaluators via team tools to ask clarifying questions, challenge scores, or request deeper analysis before writing the final output

The repo ships with default scaffold agents for Ethereum/Octant public goods, but they're a starting point — not a finished product. **The whole point is that you run `/council:setup` and reshape it into your own council.** Every time you run `/council:setup`, `/council:add-agent`, or `/council:remove-agent`, you're changing which agents exist and what they evaluate. The council you end up with is yours.

### Get started

1. Run settings (one time — this enables Claude teams, which the plugin needs for inter-agent communication):
   ```
   /council:settings
   ```

2. Try the default scaffold on a real project to see the pattern:
   ```
   /council:evaluate Aave DAO
   ```

3. Then make it yours. You have several options:

   **Redesign the whole council** for a different domain:
   ```
   /council:setup DeFi lending protocols
   /council:setup climate impact DAOs
   /council:setup developer tooling grants
   ```

   **Add or remove individual agents** to tune what gets evaluated:
   ```
   /council:add-agent
   /council:remove-agent
   ```

   **Edit agents directly** — every agent is just a markdown file in `agents/`. Change scoring dimensions, add data sources, adjust calibration. The orchestrator discovers agents by filename prefix, so there's no config to update.

   **Change what gets produced** — modify agent output templates to generate JSON, comparison matrices, grant proposals, or whatever artifact your use case needs.

Start with step 2 — run an evaluation, read the report, then decide what to change.

Then ask the user: "Want to try `/council:evaluate Aave DAO` to see the scaffold in action?"

## Overview

Claude Code plugin that generates multi-agent evaluation councils. Ships with scaffold agents for public goods — run `/council:setup` to make it yours. Three-wave pattern: data → evaluation → synthesis.

## Plugin Structure

```
agents/       Agent definitions (auto-discovered by filename prefix)
skills/       Skill definitions (orchestration, conversation, generation)
docs/         Per-skill and per-agent documentation
research/     Domain research files (generated by design/add-agent workflows)
council-out/  Runtime evaluation output (created by evaluate skill)
```

## Skills

| Skill | Invocable | Purpose |
|-------|-----------|---------|
| `council:settings` | Yes | Install the council plugin — shell alias + teams feature + permissions |
| `council:setup` | Yes | Design your evaluation council — domain, agents, lenses, data sources |
| `council:evaluate` | Yes | Evaluate a public goods project using a multi-agent council |
| `council:add-agent` | Yes | Add a new agent to the evaluation council |
| `council:remove-agent` | Yes | Remove an agent from the evaluation council |
| `council:design-conversation` | No | Sub-skill: council domain discovery |
| `council:design-agent-conversation` | No | Sub-skill: single agent design |
| `council:research-agent` | No | Sub-skill: domain research for an agent |
| `council:generate-agent` | No | Sub-skill: generate agent definition |
| `council:document-agent` | No | Sub-skill: generate agent documentation |

## Semantic Tokens

Skills use `$TOKEN` placeholders that expand from runtime context. These are not shell variables — they are semantic placeholders resolved by the executing model from conversation state, tool results, or earlier skill output.

| Token | Set by | Meaning |
|-------|--------|---------|
| `$ARGUMENTS` | Skill framework | Raw arguments passed after the skill name (e.g., `DeFi protocols` in `/council:setup DeFi protocols`) |
| `$PROJECT` | `evaluate` Step 1 | Project name or URL being evaluated |
| `$SLUG` | `evaluate` Step 1 | URL-safe slug derived from `$PROJECT` (lowercase, hyphens, max 40 chars) |
| `$DOMAIN` | `setup` Step 2 | Evaluation domain from user input or conversation |
| `$DATA_AGENTS` | `setup` Step 3 / `evaluate` Step 1 | List of data-* agent files discovered or selected |
| `$EVAL_AGENTS` | `setup` Step 3 / `evaluate` Step 1 | List of eval-* agent files discovered or selected |
| `$SYNTH_APPROACH` | `setup` Step 3 | Selected synthesis format (single chair / debate / ranked) |
| `$WAVE` | `add-agent` Step 1 | Agent wave type: `data`, `eval`, or `synth` |
| `$NAME` | `add-agent` Step 2 | Agent short name (e.g., `governance`, `audits`) |
| `$DESCRIPTION` | `add-agent` Step 2 | One-line agent description from conversation |
| `$PURPOSE` | `add-agent` Step 2 | Detailed agent purpose from conversation |
| `$SOURCES` | `add-agent` Step 2 | Data sources config (data agents) |
| `$DIMENSIONS` | `add-agent` Step 2 | Scoring dimensions config (eval agents) |
| `$RESEARCH_NEEDED` | `add-agent` Step 2 | What domain knowledge to gather before generation |
| `$TEMPLATE` | `add-agent` Step 4 | Resolved path to template agent file |
| `$AGENT` | `remove-agent` Step 1 | Selected agent name for removal |
| `$DATA_DIR` | Agent runtime | Path to Wave 1 data output: `council-out/$SLUG/data` |
| `$OUTPUT_DIR` | Agent runtime | Path to agent's output directory |
| `$EVAL_DIR` | Agent runtime | Path to Wave 2 eval output: `council-out/$SLUG/eval` |
| `$OUTPUT_PATH` | Agent runtime | Path to final report: `council-out/$SLUG/REPORT.md` |

Tokens in agent definitions (`$PROJECT`, `$DATA_DIR`, `$OUTPUT_DIR`, `$EVAL_DIR`, `$OUTPUT_PATH`) are injected by the `evaluate` orchestrator when it constructs agent prompts at runtime.

## Agent Discovery

The `evaluate` orchestrator discovers agents from the filesystem by prefix:

```
agents/data-*.md  → Wave 1 (data gathering, parallel)
agents/eval-*.md  → Wave 2 (evaluation, parallel, independent)
agents/synth-*.md → Wave 3 (synthesis)
```

No registry. No config file. Add an agent = create a markdown file with the right prefix.

## Wave Execution Model

```
Wave 1: ALL data agents spawn in a single message (parallel)
  ↓ gate: poll TaskList until all Wave 1 tasks complete
Wave 2: ALL eval agents spawn in a single message (parallel)
  ↓ gate: poll TaskList until all Wave 2 tasks complete
Wave 3: Synth agents spawn (parallel for single-chair/ranked; sub-waved for debate)
  ↓ gate: poll TaskList until all Wave 3 tasks complete
Report presented to user
```

Evaluators never see each other's scores. Independence is the point.

## Conventions

- Agent frontmatter fields: `name`, `description`, `tools`
- Agent process steps always start with `TaskUpdate` (claim) and end with `TaskUpdate` (complete) + `SendMessage` (summary)
- Eval agents score on exactly 5 dimensions, 1-10 scale, with calibration table
- Data agents include edge case handling for missing data
- All output is markdown

Additional command for deploying evaluated councils to production:

- `data-octant-scraper` — Scrape Octant Epoch 11+ project data
- `data-karma` — Pull Karma GAP scores per project
- `data-social-indexer` — Index GitHub, Farcaster, X activity
- `data-global-sources` — Aggregate DefiLlama, OSO, L2Beat data
- `eval-quantitative` — Numeric scoring (activity, funding efficiency, ecosystem impact)
- `eval-qualitative` — Narrative assessment with cited evidence
- `eval-ostrom` — Score against Elinor Ostrom's 8 Rules for Managing a Commons
- `synth-ostrom-report` — Generate Ostrom radar chart + final report
- `synth-eas-attestation` — Produce EAS SDK-compatible attestation JSON

Reference: `Ostrom-Rules.md` at repo root contains the exact 8 rules.
